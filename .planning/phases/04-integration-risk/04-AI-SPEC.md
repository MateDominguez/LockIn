## 1. System Classification

**System Type:** Multi-Agent Workflow

**Description:**
A Bayesian multi-agent investment system with 7 specialized agents orchestrated via LangGraph StateGraph. Phase 4 integrates all agents into a cohesive multi-ticker pipeline with fan-out orchestration (shared Macro Oracle → parallel per-ticker Bull⇄Bear→Strategist→Guardian→Judge → portfolio-level Optimizer). "Good" means the full pipeline produces mathematically correct Bayesian consensus, enforces Guardian veto logic, respects position sizing constraints, and surfaces a transparent audit trail for every decision.

**Critical Failure Modes:**
1. **Margin math error** — Adaptive margin of safety calculated incorrectly (wrong bounds, missing agent contributions, or unclamped values) leading to under/over-sized positions
2. **Guardian veto bypass** — High-risk assets (Z<1.1, M>-2.22, Debt/EBITDA>3x) pass through without triggering veto, exposing portfolio to distressed companies
3. **State corruption in fan-out** — Per-ticker parallel execution corrupts shared state or loses agent outputs, producing incomplete or inconsistent portfolio recommendations
4. **Bayesian synthesis error** — Judge incorrectly combines Bull/Bear probability distributions, producing nonsensical conviction scores or recommendations
5. **Position limit violation** — Optimizer ignores sector caps (35%) or concentration limits (12%), creating dangerous portfolio concentration

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Finance -- Quantitative Value Investing

**User Population:** Single operator (thesis author) using the system as a personal investment research tool for value stock analysis. The user has finance domain knowledge and acts as both developer and end-user. Not deployed commercially; no external users in v1.

**Stakes Level:** High

**Output Consequence:** The system produces BUY/HOLD/PASS recommendations with specific position sizes (percentage of portfolio). When acted upon, these outputs direct real capital allocation decisions. An incorrect recommendation -- buying a distressed company, oversizing a position, or missing a fraud signal -- results in direct financial loss. The margin of safety mechanism is the primary defense against valuation error, so incorrect margin calculation compounds downstream into every position decision.

### Known Failure Modes in This Domain

1. **Stale data producing false safety signals.** Financial data APIs (yfinance) can return cached or delayed data. If the Guardian calculates Z-Score from last quarter's balance sheet while the company has since issued significant new debt, the safety signal is misleading. This is not generic "data quality" -- it is a domain-specific risk where the system's risk assessment diverges from current reality because financial conditions change between reporting periods.

2. **Normalized earnings miscalculation in cyclical companies.** The Value Hunter normalizes earnings to remove one-time items, but for cyclical companies (e.g., commodity producers, homebuilders), "normalized" earnings during a cycle peak are artificially high. The resulting intrinsic value estimate is inflated, and the margin of safety appears adequate when it is not. Professional analysts adjust for cycle position; LLMs tend to accept reported averages at face value.

3. **Correlated position risk invisible to per-ticker analysis.** The fan-out architecture analyzes each ticker independently. If the portfolio contains 5 tickers that are all in the same supply chain or all sensitive to the same interest rate scenario, the Optimizer's sector cap (35%) may not catch the correlation. Individual position sizes look safe but the portfolio has concentrated factor exposure. This is the difference between sector diversification and true risk diversification.

4. **Conviction score inflation from LLM confidence patterns.** LLMs are trained to express confidence in their outputs. In Bull-Bear synthesis, this manifests as conviction scores that cluster in the 0.6-0.8 range regardless of actual thesis strength. A portfolio where everything is "moderate-high conviction" provides no meaningful differentiation for position sizing. Professional portfolio managers expect conviction to follow a distribution with genuine low-conviction calls.

## 2. Framework Decision

**Selected Framework:** LangGraph (keep current — no change)

**Version:** >=0.2.0 (pinned in pyproject.toml)

**Rationale:**
LangGraph is already deeply embedded across 3 completed phases with StateGraph, PostgreSQL checkpointing (langgraph-checkpoint-postgres), HITL interrupts, and conditional edges. Phase 4's multi-ticker fan-out maps directly to LangGraph's Send API or subgraph patterns. The decision matrix confirms LangGraph as the top pick for stateful multi-agent workflows with audit trails and regulated/compliance requirements (EU AI Act). Switching frameworks at Phase 4 would mean rewriting ~10 weeks of working code for zero benefit.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| asyncio.gather outer loop | Viable fallback if Send API proves too complex — would wrap independent per-ticker graph invocations. Additive, not a replacement. Keep as Plan B. |
| CrewAI | No checkpointing, coarse error handling, would require full rewrite of existing graph infrastructure |
| OpenAI Agents SDK | Vendor lock-in to OpenAI; project uses Gemini; no persistent state management |

**Vendor Lock-In Accepted:** No — LangGraph is model-agnostic. Currently using Gemini Flash via langchain-google-genai, with DeepSeek R1 identified as future paid reasoning candidate.

### Entry Point Pattern: Multi-Ticker Fan-Out with Send API
```python
"""
Multi-ticker orchestration: Macro Oracle (shared) -> fan-out per ticker
-> Optimizer (portfolio-level).

Uses Send API to dynamically spawn one per-ticker subgraph invocation
per ticker symbol. Results collected via Annotated[list, operator.add] reducer.
"""
import operator
from typing import Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send
from typing_extensions import TypedDict


class PortfolioState(TypedDict, total=False):
    """Top-level orchestrator state for multi-ticker pipeline."""
    tickers: list[str]
    macro_regime: dict                                       # shared across all tickers
    macro_confidence: float
    macro_narrative: str
    # Reducer: each per-ticker Send appends its result to this list
    ticker_results: Annotated[list[dict], operator.add]
    # Optimizer output (portfolio-level)
    optimizer_portfolio: dict
    optimizer_metrics: dict


def macro_oracle_node(state: PortfolioState) -> dict:
    """Runs once. Result shared across all tickers via state."""
    from lockin.agents.macro_oracle import macro_oracle
    # macro_oracle reads asset_ticker but for macro it's market-wide
    result = macro_oracle(state, config=None)
    return {
        "macro_regime": result["macro_regime"],
        "macro_confidence": result["macro_confidence"],
        "macro_narrative": result["macro_narrative"],
    }


def fan_out_tickers(state: PortfolioState) -> list[Send]:
    """Conditional edge: emit one Send per ticker to the per-ticker subgraph."""
    macro_ctx = {
        "macro_regime": state["macro_regime"],
        "macro_confidence": state["macro_confidence"],
        "macro_narrative": state["macro_narrative"],
    }
    return [
        Send("per_ticker_pipeline", {
            "asset_ticker": ticker,
            **macro_ctx,
        })
        for ticker in state["tickers"]
    ]


def per_ticker_pipeline(state: dict) -> dict:
    """Runs the full single-ticker graph (existing create_graph) for one ticker.

    Returns result wrapped in a list for the reducer.
    """
    from lockin.graph.builder import create_graph
    from lockin.graph.state import create_initial_state

    ticker = state["asset_ticker"]
    initial = create_initial_state(ticker)
    # Inject macro context from shared state
    initial["macro_regime"] = state.get("macro_regime", {})
    initial["macro_confidence"] = state.get("macro_confidence", 0.5)
    initial["macro_narrative"] = state.get("macro_narrative", "")

    graph = create_graph()  # no checkpointer for inner graph
    result = graph.invoke(initial)
    return {"ticker_results": [{"ticker": ticker, **result}]}


def portfolio_optimizer_node(state: PortfolioState) -> dict:
    """Runs once after all per-ticker pipelines complete."""
    # Aggregate all ticker_results into portfolio construction
    from lockin.agents.optimizer import optimize_portfolio
    return optimize_portfolio(state["ticker_results"])


# --- Build orchestrator graph ---
builder = StateGraph(PortfolioState)
builder.add_node("macro_oracle", macro_oracle_node)
builder.add_node("per_ticker_pipeline", per_ticker_pipeline)
builder.add_node("portfolio_optimizer", portfolio_optimizer_node)

builder.add_edge(START, "macro_oracle")
builder.add_conditional_edges("macro_oracle", fan_out_tickers, ["per_ticker_pipeline"])
builder.add_edge("per_ticker_pipeline", "portfolio_optimizer")
builder.add_edge("portfolio_optimizer", END)

orchestrator = builder.compile()

# --- Invoke ---
result = orchestrator.invoke({
    "tickers": ["AAPL", "MSFT", "GOOGL", "AMZN", "NVDA"],
})
```

### Key Abstractions (Phase 4 focus)

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `Send(node_name, state)` | Dynamically spawns a node invocation with custom state. Returns from a conditional edge function as a list. All Sends to the same node run in parallel. | Fan-out: one `Send` per ticker to `per_ticker_pipeline`. The graph waits for ALL Sends to complete before continuing to the next edge. |
| `Annotated[list, operator.add]` | State reducer that concatenates lists instead of overwriting. Each node return appends to the accumulated list. | Collecting `ticker_results` from parallel per-ticker pipelines. Without the reducer, only the last ticker's result would survive. |
| Subgraph (compiled graph as node) | A compiled `StateGraph` passed as a node to a parent graph. Shares state keys automatically if TypedDict fields match. Checkpointer only needed at parent level. | Alternative to Send API when per-ticker pipeline needs its own checkpointing or when state schemas differ between parent and child. |
| `graph.ainvoke()` / `graph.astream()` | Async equivalents of `invoke`/`stream`. Required for concurrent LLM calls with semaphore-based rate limiting. | Multi-ticker execution where you want `asyncio.Semaphore` to throttle concurrent Gemini API calls (default: 2 concurrent). |
| `Command(update, goto, graph=Command.PARENT)` | Allows a subgraph node to update parent state AND route to a specific parent node. Useful for early termination (e.g., Guardian veto in a subgraph). | If using subgraph pattern instead of Send: Guardian veto can propagate up to parent graph without completing remaining per-ticker nodes. |

### Common Pitfalls

1. **Send state is NOT the parent state.** Each `Send("node", {...})` creates an isolated state for that node invocation. The node receives ONLY what you pass in the Send dict, not the full parent `PortfolioState`. If you forget to include `macro_regime` in the Send payload, the per-ticker pipeline will not have macro context. Always explicitly pass all required fields.

2. **Missing reducer causes silent data loss.** If `ticker_results` is typed as `list[dict]` without `Annotated[..., operator.add]`, only the LAST Send's return value overwrites the field. With 5 tickers, you lose 4 results silently. Every field that collects from parallel Sends MUST have a reducer.

3. **Subgraph checkpointer inheritance is implicit.** When you add a compiled subgraph as a node, the parent's checkpointer is used automatically -- do NOT compile the subgraph with its own checkpointer, or you get double-checkpointing and state conflicts. Only the top-level `orchestrator.compile(checkpointer=...)` should receive the checkpointer.

4. **`asyncio.run()` inside a running event loop crashes.** If the orchestrator is invoked from Streamlit or Jupyter (which already have an event loop), calling `asyncio.run(orchestrator.ainvoke(...))` raises `RuntimeError: This event loop is already running`. Use `await orchestrator.ainvoke(...)` directly, or use `asyncio.get_event_loop().run_until_complete(...)` only from synchronous entry points with no existing loop.

5. **Send does not guarantee execution order.** Sends to the same node run concurrently and may complete in any order. Do not assume `ticker_results[0]` corresponds to `tickers[0]`. Each result must carry its own `ticker` identifier for correct downstream aggregation in the Optimizer.

*Pattern A: Send API (recommended)*
```python
# orchestrator.py
from langgraph.types import Send

def fan_out_tickers(state: PortfolioState) -> list[Send]:
    """One Send per ticker. LangGraph runs all in parallel, collects via reducer."""
    macro = {k: state[k] for k in ("macro_regime", "macro_confidence", "macro_narrative")}
    return [Send("per_ticker", {"asset_ticker": t, **macro}) for t in state["tickers"]]

# per_ticker node invokes the existing single-ticker graph internally
def per_ticker(state: dict) -> dict:
    graph = create_graph()  # no checkpointer -- inner graph is stateless
    result = graph.invoke(create_initial_state(state["asset_ticker"]) | state)
    return {"ticker_results": [{"ticker": state["asset_ticker"], **result}]}
```

*Pattern B: asyncio.gather fallback (Plan B per D-04)*
```python
# If Send API proves too complex or checkpointing conflicts arise
async def run_multi_ticker(tickers: list[str], macro_ctx: dict) -> list[dict]:
    sem = asyncio.Semaphore(2)  # D-04: max 2 concurrent LLM calls

    async def run_one(ticker: str) -> dict:
        async with sem:
            graph = create_graph()
            initial = create_initial_state(ticker) | macro_ctx
            return await graph.ainvoke(initial)

    return await asyncio.gather(*[run_one(t) for t in tickers])
```

**Rate Limiting:**
```python
# utils/rate_limiter.py
import asyncio
from functools import wraps

class GeminiRateLimiter:
    """Semaphore-based rate limiter for concurrent LLM calls.

    Default: 2 concurrent calls (D-04). Gemini Flash free tier = 500 RPD.
    For 5 tickers * ~8 LLM calls each = ~41 calls per portfolio run.
    """
    def __init__(self, max_concurrent: int = 2):
        self._sem = asyncio.Semaphore(max_concurrent)

    async def __aenter__(self):
        await self._sem.acquire()
        return self

    async def __aexit__(self, *exc):
        self._sem.release()
```

**State Management:**

Phase 4 introduces a two-level state architecture:

1. **`PortfolioState`** (new) -- Orchestrator-level state. Contains `tickers: list[str]`, shared macro context, and `ticker_results: Annotated[list[dict], operator.add]` with a list reducer. Lives in `graph/portfolio_state.py`.

2. **`InvestmentState`** (existing) -- Per-ticker state. Unchanged. Each `Send` invocation creates an isolated `InvestmentState` for one ticker. The per-ticker pipeline returns its result as a single-element list that the reducer concatenates.

3. **Checkpointing:** PostgreSQL checkpointer attached ONLY to the orchestrator graph. Inner per-ticker graphs run without checkpointing (they are effectively pure functions given macro context + ticker). If a run fails mid-way, the orchestrator checkpoint restores the last completed state (e.g., macro_oracle done, 3/5 tickers done) and re-runs only the remaining tickers.

4. **State isolation guarantee:** Send API creates independent state copies per ticker. No shared mutable state between concurrent per-ticker pipelines. The only merge point is the `ticker_results` reducer at the `portfolio_optimizer` node.

**Context Window Strategy:**

Gemini 2.5 Flash has a 1M token context window -- context window is not the bottleneck. The constraint is output quality degradation with overly long prompts.

| Agent | Context Strategy |
|-------|-----------------|
| Macro Oracle | Summarized macro indicators (~2K tokens). Runs once, shared. |
| Value Hunter | Earnings transcript truncated to most recent quarter (~8K tokens). Financial statements summarized. |
| Bear | Receives bull_thesis (~1K) + financial data. Does NOT see bull_defense to avoid anchoring (Phase 3 decision). |
| Strategist | VeTO score + earnings sentiment. Compact signals, not raw transcripts. |
| Guardian | Financial ratios only (~500 tokens). No narrative -- pure quantitative screening. |
| Judge | Receives all modifier outputs + distributions. Heaviest context (~4K tokens) but well within limits. |
| Optimizer | Receives JudgeOutput per ticker. For 5 tickers, ~2K tokens total. |

No truncation or summarization needed at current scale (5-10 tickers). Revisit if ticker count exceeds 50 (unlikely for personal portfolio use case).

**Pydantic model for portfolio-level validation:**
```python
from pydantic import BaseModel, Field, field_validator

class TickerResult(BaseModel):
    """Validated output from a single-ticker pipeline run."""
    ticker: str
    recommendation: str  # BUY | HOLD | PASS
    conviction: float = Field(ge=0.0, le=1.0)
    margin_of_safety: float = Field(ge=0.15, le=0.60)  # D-11 bounds
    position_size: float = Field(ge=0.0, le=0.10)
    veto_score: float = Field(ge=0.0, le=1.0)
    margin_breakdown: dict = Field(default_factory=dict)

    @field_validator("recommendation")
    @classmethod
    def valid_recommendation(cls, v: str) -> str:
        if v not in ("BUY", "HOLD", "PASS"):
            raise ValueError(f"Invalid recommendation: {v}. Must be BUY, HOLD, or PASS.")
        return v

    @field_validator("margin_breakdown")
    @classmethod
    def has_required_keys(cls, v: dict) -> dict:
        required = {"base", "oracle", "guardian", "strategist", "raw_total", "clamped"}
        missing = required - set(v.keys())
        if missing:
            raise ValueError(f"margin_breakdown missing keys: {missing}")
        return v


class PortfolioResult(BaseModel):
    """Validated output from the full multi-ticker orchestrator."""
    ticker_results: list[TickerResult]
    total_allocation: float = Field(ge=0.0, le=1.0)
    max_sector_weight: float = Field(le=0.35)  # Sector cap
    max_position_weight: float = Field(le=0.12)  # Concentration limit
```

**Integration approach -- post-hoc validation (not `.with_structured_output()`):**

LockIn agents use manual LLM calls via `get_llm().invoke()` with structured prompts, then parse responses into dataclasses. The agents do NOT use LangChain's `.with_structured_output()` because the output contracts (`ConfidenceModifier`, `JudgeOutput`) involve computed fields (Bayesian math, margin calculations) that are NOT LLM-generated.

Validation strategy:
```python
# In portfolio_optimizer or orchestrator post-processing:
from pydantic import ValidationError

def validate_pipeline_output(raw_result: dict) -> TickerResult:
    """Validate a single-ticker pipeline result. Raises on contract violation."""
    try:
        return TickerResult(**raw_result)
    except ValidationError as e:
        # Log the validation failure with full context for debugging
        logger.error("Contract violation for %s: %s", raw_result.get("ticker"), e)
        raise  # Do NOT retry -- this is a math/logic bug, not a transient error

# Retry logic: NOT applicable here. Unlike LLM parsing errors where retrying
# with a corrective prompt makes sense, contract violations in LockIn indicate
# bugs in agent math (e.g., margin outside [0.15, 0.60]). These must fail loud.
```

### 4b.2 Async-First Design

**How async works in LangGraph:**
LangGraph's `StateGraph` supports both sync (`invoke`, `stream`) and async (`ainvoke`, `astream`) execution. Nodes can be either sync or async functions -- LangGraph wraps sync nodes in `asyncio.to_thread()` automatically when using `ainvoke`.

**Current state:** All 7 agents are synchronous (using `llm.invoke()` not `llm.ainvoke()`). This is fine for single-ticker execution. For multi-ticker fan-out, the Send API handles parallelism at the graph level -- individual nodes do not need to be async.

**The one common mistake:**
```python
# WRONG: calling asyncio.run() from Streamlit or Jupyter
result = asyncio.run(orchestrator.ainvoke({"tickers": [...]}))
# RuntimeError: This event loop is already running

# CORRECT: from synchronous entry point (CLI, script)
result = orchestrator.invoke({"tickers": [...]})  # sync is fine

# CORRECT: from async context (Streamlit with async mode, FastAPI)
result = await orchestrator.ainvoke({"tickers": [...]})
```

**When to use sync vs async:**
- **`invoke()` (sync):** Default for Phase 4. Single-ticker pipeline, CLI usage, tests. Simpler to debug.
- **`ainvoke()` (async):** Required only when wrapping with `asyncio.Semaphore` for rate limiting in Pattern B (asyncio.gather fallback). Send API handles parallelism without requiring async nodes.
- **`astream()` (async streaming):** For Streamlit dashboard (Phase 5+). Stream per-ticker progress updates to the UI as each ticker completes.

**Rate limiter integration with async nodes:**
```python
# If converting agents to async for Pattern B:
async def value_hunter_async(state: InvestmentState, config: RunnableConfig) -> dict:
    async with rate_limiter:  # GeminiRateLimiter(max_concurrent=2)
        llm = get_llm()
        # langchain-google-genai supports ainvoke natively
        response = await llm.ainvoke(messages)
        # ... rest of agent logic
```

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation:**

All 7 agents already follow a consistent pattern from Phase 3:
```python
# System prompt: role definition, constraints, output format
SYSTEM_PROMPT = """You are the Bear Analyst in an investment research team.
Your role is to challenge the Bull thesis with data-driven counterarguments.
You must NOT see the Bull's defense -- evaluate independently.
Output format: JSON with keys: challenges, red_flags, bear_thesis, conviction."""

# User prompt: dynamic data injection
user_prompt = f"""Analyze {ticker} given these financials:
{financial_summary}

Bull thesis to challenge:
{bull_thesis}

Respond in the specified JSON format."""
```

**Phase 4 prompt changes are minimal:**
- VeTO margin wiring is pure Python (no prompt change needed)
- Margin bounds [0.15, 0.60] are constants in `judge_math.py` (no prompt change)
- Optimizer portfolio aggregation may need a prompt update to handle multi-ticker context

**Few-shot guidance:**
Not used in LockIn agents. Each agent receives structured financial data and produces structured output. The output format is enforced by post-processing (regex/JSON parsing), not by few-shot examples. Adding few-shot examples would consume context budget without improving structured output quality for this domain.

**Token budget strategy:**
- `max_tokens` is NOT explicitly set in `get_llm()` -- Gemini defaults to 8192 output tokens.
- For Phase 4, this is acceptable: no agent produces outputs longer than ~2K tokens.
- If adding explicit limits: set `max_tokens=4096` for modifier agents (short structured output), `max_tokens=8192` for Value Hunter and Judge (longer narratives).
- Production TODO: add explicit `max_tokens` to `get_llm()` to prevent runaway token usage on malformed prompts.

### 4b.4 Context Window Management

**Multi-agent context strategy for LockIn:**

The key insight: each agent sees only its required inputs, not the full pipeline state. This is by design -- agents are specialists, not generalists.

| Pattern | Implementation |
|---------|---------------|
| **Shared context (Macro Oracle)** | Runs once. Its output (~500 tokens) is injected into every per-ticker pipeline via Send state. Compact enough that no summarization is needed. |
| **Dialectic context (Bull-Bear)** | Bear sees bull_thesis but NOT bull_defense (Phase 3 decision). Value Hunter sees bear_challenges for rebuttal. Jaccard similarity detects convergence to terminate. Context grows linearly with iterations (max 2 rounds = ~4K tokens). |
| **Accumulated modifiers (Judge)** | Judge receives 3 `ConfidenceModifier` objects (~200 tokens each) + 2 `ValueDistribution` objects (~300 tokens each). Total Judge context: ~1.5K tokens of structured data + ~1K system prompt. Well within limits. |
| **Portfolio aggregation (Optimizer)** | Receives `JudgeOutput` per ticker. At 5 tickers, ~2.5K tokens. At 20 tickers, ~10K tokens. No summarization needed until 50+ tickers. |

**When context becomes a problem (future):**
- If ticker count exceeds 20: summarize each `JudgeOutput` to key fields only (recommendation, conviction, margin, position_size) before passing to Optimizer.
- If earnings transcripts grow: the existing `_fetch_earningscall_transcript()` already returns the most recent quarter only. Further truncation: extract only MD&A and guidance sections.

## 5. Evaluation Strategy

> Written by `gsd-eval-planner`. Grounded in Section 1b domain rubric ingredients and Section 1 critical failure modes.

Phase 4 is an integration phase -- the primary risk is agents working incorrectly *together*, not individual agent quality (tested in Phase 3). Eval dimensions are therefore focused on mathematical correctness at the pipeline level, state integrity across fan-out, and constraint enforcement. Most dimensions are code-testable because the critical behaviors are deterministic math and structural contracts, not subjective LLM output quality.

### Dimensions

| # | Dimension | Rubric | Measurement | Priority |
|---|-----------|--------|-------------|----------|
| E1 | **Margin math correctness** | PASS: `margin_of_safety` within [0.15, 0.60] for all tickers. `margin_breakdown` contains all 6 required keys (`base`, `oracle`, `guardian`, `strategist`, `raw_total`, `clamped`). `base + oracle + guardian + strategist == raw_total`. `clamped == clamp(raw_total, 0.15, 0.60)`. VeTO thresholds produce correct adjustments: VeTO < 0.3 yields +0.10, VeTO < 0.4 yields +0.05, VeTO 0.4-0.7 yields 0.00, VeTO > 0.7 yields -0.03, VeTO > 0.85 yields -0.05. FAIL: Any margin outside bounds, missing breakdown keys, or arithmetic mismatch between components and total. | **Code** -- Pydantic `TickerResult` validator + dedicated `test_contracts.py` assertions. Zero LLM calls. | Critical |
| E2 | **Guardian veto enforcement** | PASS: Every ticker with Z-Score < 1.1, Beneish M > -2.22, or Debt/EBITDA > 3x receives `recommendation = PASS` (hard veto) or triggers HITL. No BUY recommendation ever issued for a vetoed ticker. Veto reason logged in audit trail with specific threshold values. FAIL: A BUY recommendation for a ticker that meets any veto threshold. Guardian output present in state but ignored by downstream Judge/Optimizer. | **Code** -- Contract test with synthetic Guardian outputs at threshold boundaries (Z=1.09, Z=1.11, M=-2.21, M=-2.23). Verify Judge output respects veto flag. | Critical |
| E3 | **State integrity across fan-out** | PASS: For N tickers, `len(ticker_results) == N`. Each result contains a unique `ticker` field matching one input ticker. No duplicate tickers. No missing agent outputs within any ticker result (all 7 agent outputs present). Macro context (`macro_regime`, `macro_confidence`) identical across all ticker results. FAIL: `len(ticker_results) < N` (silent data loss from missing reducer). Ticker results missing agent fields. Macro context differs between tickers (state leakage). | **Code** -- Contract test invoking orchestrator with 3 tickers using mock agents. Assert result count, field completeness, and macro context equality. | Critical |
| E4 | **Position sizing discipline** | PASS: Every `position_size` in [0.0, 0.10]. No single position exceeds 12% of portfolio (`max_position_weight <= 0.12`). No sector exceeds 35% of total allocation (`max_sector_weight <= 0.35`). `total_allocation` in [0.0, 1.0]. Kelly fraction uses Kelly/3 (not full Kelly). Position sizes decrease when conviction < 0.5 or macro regime is contractionary. FAIL: Any position exceeding limits. Sector cap breached. Total allocation > 100%. Full Kelly applied. | **Code** -- Pydantic `PortfolioResult` validator. Contract tests with 5 same-sector tickers to verify sector cap enforcement. Test with high-conviction inputs to verify concentration limit holds. | Critical |
| E5 | **Bayesian synthesis correctness** | PASS: Judge conviction score in [0.0, 1.0]. When Bear presents strong objections (high-conviction Bear), Judge conviction decreases compared to weak-Bear baseline. When all modifiers signal caution (high margin, contractionary macro), recommendation trends toward HOLD/PASS rather than BUY. FAIL: Conviction is constant regardless of Bear strength (clustering in 0.6-0.8). Judge always sides with Bull. Conviction outside [0.0, 1.0]. | **Code** -- Contract test comparing Judge output with strong-Bear vs weak-Bear modifier inputs. Verify conviction ordering: `conviction(weak_bear) > conviction(strong_bear)`. | Critical |
| E6 | **Margin of safety transparency** | PASS: Every pipeline output includes `margin_breakdown` with individually visible contributions from Oracle, Guardian, and Strategist. Each contribution is a float with clear sign. User can reconstruct final margin from components: `base + oracle + guardian + strategist == raw_total`. FAIL: `margin_breakdown` is empty, missing keys, or contains only the final clamped value without component decomposition. | **Code** -- Pydantic `margin_breakdown` validator (6 required keys). Arithmetic consistency check in contract tests. | High |
| E7 | **Bull-Bear dialectical quality** | PASS: Bear thesis contains company-specific evidence (named competitors, specific financial metrics, concrete risk factors). Bear objections differ substantively from Bull thesis (not just negations). Bull rebuttal addresses at least one specific Bear objection. Judge rationale references arguments from both sides. FAIL: Bear produces only generic objections ("valuation is high", "macro risks exist"). Bull rebuttal ignores Bear entirely. Judge rationale mentions only Bull arguments. | **LLM Judge** -- Gemini Flash evaluates Bear/Bull/Judge outputs against a rubric prompt. Run on smoke test outputs only (not CI). Calibrate against thesis author judgment on 5 examples before trusting. | High |
| E8 | **Pipeline task completion** | PASS: Full multi-ticker pipeline completes without unhandled exceptions. Every ticker produces a valid `TickerResult`. Optimizer produces a valid `PortfolioResult`. Total wall-clock time < 120 seconds for 5 tickers. FAIL: Any unhandled exception. Any ticker missing from results. Optimizer fails to produce output. Pipeline hangs or exceeds 180 seconds. | **Code** -- Live smoke test (`@pytest.mark.slow`). Asserts structural completeness and timing. | High |

## 6. Guardrails

> Written by `gsd-eval-planner`. Guardrails are classified by the question: "If this goes wrong, is it catastrophic?"

### Online (Real-Time)

These guardrails run on every pipeline invocation. They are all code-based checks (no LLM calls) to avoid adding latency or consuming the 500 RPD budget.

| Guardrail | Trigger | Intervention | Implementation |
|-----------|---------|--------------|----------------|
| **Guardian veto gate** | `fraud_veto == True` OR Z-Score < 1.1 OR M-Score > -2.22 OR Debt/EBITDA > 3x | **Block** -- force `recommendation = PASS`. Log veto reason to audit trail. Trigger HITL if configured. | Already exists in `guardian.py`. Phase 4 verifies Judge respects the flag (contract test E2). |
| **Margin bounds clamp** | `margin_of_safety` computed outside [0.15, 0.60] | **Clamp** -- force value to nearest bound. Log pre-clamp value in `margin_breakdown["raw_total"]` for transparency. | Already exists in `judge_math.py`. Phase 4 updates constants from [0.20, 0.70] to [0.15, 0.60] (D-11). |
| **Position limit enforcer** | Any `position_size > 0.10` OR any sector weight > 0.35 OR `total_allocation > 1.0` | **Cap** -- reduce position sizes proportionally until limits satisfied. Log original vs capped values. | Exists in `optimizer.py`. Phase 4 adds portfolio-level aggregation check across tickers (new). |
| **Pydantic output validator** | `TickerResult` or `PortfolioResult` validation fails | **Block** -- raise `ValidationError`, do not return partial/invalid results. Log full validation error context. | New in Phase 4. Post-hoc validation at orchestrator output (Section 4b.1). |
| **Fan-out completeness check** | `len(ticker_results) != len(tickers)` after all Sends complete | **Block** -- raise error, do not proceed to Optimizer with incomplete data. Log which tickers are missing. | New in Phase 4. Check in `portfolio_optimizer_node` before aggregation. |

### Offline (Flywheel)

These run as periodic batch analysis, not on every request. Appropriate for a single-operator personal tool where every pipeline run is already reviewed manually during paper trading.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Conviction score distribution** | Every pipeline run (logged to audit trail). Review weekly during paper trading. | If conviction clusters in 0.6-0.8 for >80% of tickers across 3+ runs, investigate LLM confidence inflation (known failure mode 4 from Section 1b). Consider adding conviction calibration or prompt adjustment to Bear/Judge. |
| **Margin adjustment frequency** | Every pipeline run. Aggregate monthly. | If Guardian/Strategist adjustments are always 0.0 (neutral), the modifiers may not be contributing meaningful signal. If always at max penalty, thresholds may be too aggressive for the ticker universe being analyzed. Either case warrants threshold review per D-10. |
| **Veto rate** | Every pipeline run. Track veto count / total tickers analyzed. | If veto rate > 50% over a month, thresholds may be too aggressive. If veto rate = 0% over 20+ tickers, Guardian may not be receiving correct financial data (stale data failure mode from Section 1b). |
| **Bull-Bear dialectical quality** | Sample 1 in 5 pipeline runs. Thesis author reads Bear thesis and Bull rebuttal for one ticker. | If Bear produces generic objections in 3+ consecutive samples, review Bear prompt for specificity requirements. If Bull never addresses Bear objections, review Bull rebuttal prompt. |
| **Pipeline latency trend** | Every pipeline run (wall-clock time logged). | If 5-ticker pipeline exceeds 120 seconds consistently, investigate rate limiting or API latency. If single-ticker exceeds 45 seconds, check for retry loops or stalled LLM calls. |

## 7. Production Monitoring

> Written by `gsd-eval-planner`. "Production" for LockIn means paper trading with a single operator, not a multi-user deployment.

**Tracing Tool:** Arize Phoenix (open-source, self-hosted, localhost:6006)

Phoenix is selected because: (1) open-source and free -- matches the project's zero-budget constraint, (2) LangChain/LangGraph native instrumentation via OpenInference, (3) trace visualization for multi-agent fan-out debugging -- each ticker's agent chain visible as nested spans, (4) no platform account or cloud dependency required.

Phoenix is **development-time tooling**, not a persistent production service. The operator starts it with `LOCKIN_TRACE=true` when debugging a specific pipeline run or investigating unexpected output. It is not always-on.

**Key Metrics to Track:**

| Metric | Source | Target |
|--------|--------|--------|
| **Pipeline completion rate** | Audit trail (PostgreSQL `audit_logs` table) | 100% -- every invocation should complete or fail with a clear error, never hang silently |
| **Margin bounds adherence** | Pydantic `TickerResult` validator (online guardrail) | 100% -- no output with margin outside [0.15, 0.60] ever reaches the operator |
| **Guardian veto accuracy** | Contract tests (CI) + manual review of vetoed/non-vetoed tickers during paper trading | 100% for known-distress tickers (zero false negatives on veto). False positive rate acceptable up to ~20%. |
| **LLM calls per portfolio run** | Simple counter in `get_llm()` wrapper (or Phoenix trace spans when enabled) | Target ~41 for 5 tickers. Alert if > 60 (indicates retry loops or unexpected dialectic iterations) |
| **Wall-clock latency** | Timer in orchestrator `invoke()` wrapper | < 120s for 5-ticker portfolio. Log warning at 90s. |

**Alert Thresholds:**

This is a personal tool run by a single operator, not a 24/7 service. "Alerts" are Python log messages and pytest failures, not PagerDuty notifications.

| Condition | Severity | Action |
|-----------|----------|--------|
| Any `ValidationError` from Pydantic output validator | ERROR | Pipeline fails loud. Fix the math/logic bug before the next run. Do not suppress. |
| Guardian veto bypassed (BUY issued for a vetoed ticker) | CRITICAL | Should never happen if contract tests pass. If it does, halt all pipeline runs and fix immediately -- this is the highest-stakes failure. |
| `len(ticker_results) < len(tickers)` | ERROR | Fan-out data loss. Check reducer configuration (`Annotated[list, operator.add]`). Do not proceed to Optimizer. |
| Pipeline wall-clock > 180s for 5 tickers | WARNING | Check for Gemini 429 rate limit errors, stalled API calls, or unbounded dialectic loop iterations. |
| LLM calls per run > 60 | WARNING | Unexpected retry behavior or additional dialectic rounds beyond the max-2 configured. Review agent logs for retry patterns. |
| Gemini 429 errors > 3 per single run | WARNING | Approaching daily rate limit. Reduce concurrency semaphore from 2 to 1, or defer remaining tickers to the next day. |

**Smart Sampling Strategy:**

Given the single-operator context and a maximum of ~12 portfolio runs per day on the free tier, the sampling strategy is straightforward:

1. **Every run (automated, zero cost):** Pydantic validation, margin bounds check, veto enforcement, fan-out completeness. All code-based guardrails from Section 6 run automatically.
2. **Every run during paper trading (human):** Thesis author reviews Optimizer output -- recommendation, position sizes, margin breakdown -- before any simulated trade execution. This is the HITL loop already designed into the system (CORE-04).
3. **1 in 5 runs (human, ~5 minutes):** Thesis author reads the full Bear thesis and Bull rebuttal for one randomly selected ticker to assess dialectical quality (E7). Look for: company-specific evidence in Bear, direct rebuttal of Bear points by Bull, and Judge synthesis that references both sides.
4. **Monthly regression (automated + human):** Run the 2 known-outcome historical tickers (one failure case, one success case) through the pipeline. Compare conviction, margin, and recommendation to previous month's run. Significant drift indicates prompt degradation, data source changes, or Gemini model updates.
5. **On-demand tracing:** Enable Phoenix with `LOCKIN_TRACE=true` when debugging a specific pipeline failure or investigating unexpected output. Inspect the nested span tree to identify which agent in which ticker's pipeline produced the anomaly.
